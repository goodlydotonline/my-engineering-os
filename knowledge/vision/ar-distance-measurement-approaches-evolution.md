# FireCopilot 测距功能方案演进史

> 记录 FireCopilot（消防隐患识别 Demo）测距功能从最早的「单目像素估算」一路演进到当前「ARCore 锁定平面 + 屏幕射线投影」的全过程。共 6 个阶段，逐一说明：**设计 / 数学原理（配图）/ 用户怎么用·能测什么·哪种最准 / 优点 / 劣势 / 为什么被下一代取代**。

---

## 0. 先理解测距的本质（贯穿所有方案）

任何「量两点间真实长度」的功能，本质都是两步：

1. **把屏幕上指定的点，还原成真实世界的三维坐标**（这一步最难，决定准不准）；
2. **对两个世界坐标求直线距离** `d = √(Δx² + Δy² + Δz²)`（这一步是固定公式，几乎不会错）。

所以整部演进史的主线，就是**第 1 步越做越可靠**：从「用像素比例猜深度」→「用 ARCore 射线命中平面」→「约束两点必须可比」→「把第二点投影到第一点锁定的平面」。

ARCore 世界坐标系：单位**米**，Y 轴朝上，原点是 session 启动时相机的位置。一个 `Anchor`（锚点）就是 ARCore 持续跟踪的一个世界坐标点。

> 关键认知（来自 [辨析：安卓手机 AR 硬件支持、ARCore、ARCore Depth API](knowledge/vision/android-ar-hardware-arcore-depth-api.md)）：普通单目安卓手机 + ARCore，短距离良好条件下能到厘米级**演示**效果，但不应承诺稳定 <1~2cm。精度由硬件（是否有 ToF/Depth）、环境（光照/纹理）、用户行为（移动快慢）、算法（平面拟合/锚点漂移）四层共同决定。

## 方案一：bbox 像素比例估算 + 手动校准（Step 6 / 4.6）

📅 2026-06-22 ｜ 非 ARCore，纯单目 ｜ 代码：`lib/core/distance/distance_estimator.dart`

### 设计
完全不依赖 AR。复用 AI 检测框（bounding box）：每类消防隐患预设一个经验「参考高度」`referenceHeight`（如灭火器约 0.5m），再用检测框在画面里的归一化高度 `bbox.height` 反推「相机到该物体的距离」。用户可输入真实距离做**手动校准**，把校准比例按隐患类型存进 `SharedPreferences`。

### 数学原理
简化的**针孔相机相似三角形**模型：

```
       真实物体
         ┃■┃ H (真实高度，用 referenceHeight 近似)
         ┃ ┃
 相机     ┃ ┃
  ◇───────┃ ┃         针孔模型: d = f · H / h
   \  f   ┃ ┃         (f=焦距, h=成像高度)
    \─────┃ ┃
     \    ┃ ┃   本方案把 f 和单位一起吸收进 referenceHeight 常数，简化为:
      \   ┃ ┃
       d (待求距离)        d ≈ referenceHeight / bbox.height
```

- `rawDistance = referenceHeight / bbox.height`（`bbox.height` 是 0~1 归一化值）；
- 校准：`distance = rawDistance × calibrationScale`，再 `clamp[0.5m, 20m]`；
- 可信度：相机流 = medium，静态照片 = low。

### 用户怎么用 · 能测什么 · 哪种最准
- **用法**：不需要任何额外操作——AI 识别到隐患后，距离标签自动挂在检测框下方。可选「校准距离」输入一次真值。
- **测什么**：**相机到物体的深度**（一个距离数字），**不是物体本身的长度/宽度**。
- **哪种最准**：物体真实高度接近预设 `referenceHeight`、且正对相机无遮挡时最接近；校准后对同类物体略好。
- **📱 手机怎么动**：**没有「两个锚点」概念，是单帧深度估算**。端稳手机、让目标完整且正对地落在画面里即可，**不需要移动到第二个点**。手机晃 → 检测框变 → 距离跳；离远离近也会直接改变估算值（这正是它不准的根源之一）。

### 优点
- 零硬件门槛，任何安卓机都能跑，不依赖 ARCore。
- 与 AI 检测天然融合，零额外交互。
- 有校准兜底，能对特定场景手动修偏。

### 劣势
- **本质是猜**：`referenceHeight` 是经验常数，物体实际高度一变就错。
- 只能给「深度」，**无法测两点间长度**（量不了通道净宽、门高这类核心消防指标）。
- 检测框抖动 → 距离抖动；忽略相机内参、姿态、透视畸变。

### 为什么被取代
消防场景真正要的是**通道净宽、安全出口宽度、防火门高度**这类**两点间实测长度**，像素估算根本给不了。→ 引入真正的空间测距（方案二）。

## 方案二：ARCore 两点 hitTest 测距 MVP（Step 7）

📅 2026-06-23 ｜ 原生 Android ARCore 首次接入 ｜ `ar/ArMeasurementActivity.kt` 等

### 设计
新建原生 ARCore Activity：`GLSurfaceView` + 自定义 `ArBackgroundRenderer` 显示相机流，屏幕中心固定一个准星。用户点两次 `+`，**每次对屏幕中心做一次 `hitTest`**，把命中点各创建成一个 `Anchor`，再求两 Anchor 的 3D 欧氏距离。`PixelCopy` 截图存证。通过 `MethodChannel`（`com.firecopilot.app/arcore_measurement`）与 Flutter 通信。

> 配套 Step 7.1 做了「相机式」UI（中心准星 + 底部 3 图标）、`AnchorProjector`（世界坐标→屏幕像素，用于画锚点/连线）、EMA 屏幕平滑，以及一连串稳定性修复（NPE、距离冻结/解冻反复横跳、过严门控回退）。

### 数学原理
`hitTest(cx, cy)` 从相机沿准星方向发射射线，与 ARCore 重建的几何（平面/特征点）求交，返回世界坐标命中点；两点直接求欧氏距离：

```
   相机 ◇
        /│\
       / │ \   两次点击，各发一条屏幕中心射线
      /  │  \
  ───A───┼───B───  ARCore 平面 / 特征点
      \      /
       d = ‖A − B‖ = √(Δx²+Δy²+Δz²)   (世界坐标，单位米)
```

### 用户怎么用 · 能测什么 · 哪种最准
- **用法**：进 AR 页 → 缓慢移动手机让 ARCore 建图 → 准星对准点 A 点 `+` → 移到点 B 点 `+` → 出距离。
- **测什么**：空间中任意两点的真实距离（理论上）。
- **哪种最准**：纹理丰富的平面上、近距离（<1m）、两点同处一个稳定平面时。
- **📱 手机怎么动**：两个点都要 `hitTest` **真实命中平面**，所以要把准星**实际对到第二端点的物理位置**。**推荐小幅平移**让准星滑过去，全程缓慢保持 TRACKING。**不建议原地纯旋转**——单目 ARCore 靠平移产生的视差维持尺度，纯旋转会退化、尺度漂移；快速大幅移动则丢跟踪。

### 优点
- 第一次实现**真实 6DoF 空间测距**，量的是世界坐标实距，原理上远胜像素估算。
- 测距与单位换算正确（后续 A4 实测 ≈29.7cm 验证了这一点）。

### 劣势
- `hitTest` 命中**不挑**：可能命中漂移的 FeaturePoint、可能两点落在**不同**平面、可能被无限平面延伸误命中。
- 锚点漂移、距离数值每帧跳变（早期在「每帧重算」与「冻结」间反复）。
- 无任何「两点是否可比」的约束 → 数值可能莫名其妙。

### 为什么被取代
命中太自由导致结果不可信。→ 收紧命中策略，只认稳定平面（方案三）。

## 方案三：救援版——仅平面命中 + 距离冻结（4.7 rescue）

📅 2026-06-24 ｜ 只改 `ArMeasurementActivity.kt` 一个文件

### 设计
状态机 `IDLE → START_PLACED → END_PLACED → SAVING` 不变。每帧只做三件事：渲染相机背景 + 把**已有**锚点投影到屏幕（含 EMA 平滑）+ 准星亮度反馈，**绝不每帧重建锚点**。点 `+` 时 `findValidPlaneHit` 只接受 `Plane && TRACKING && isPoseInPolygon(hitPose)`，**拒绝 FeaturePoint、拒绝无限平面延伸命中**。放锚前先校验相机 `TRACKING`。距离**创建即冻结**到 `finalDistanceMeters`，显示与保存同源。

### 数学原理
距离公式不变（3D 欧氏距离），关键是**命中过滤**：只接受落在平面**多边形内**的点。

```
   命中点必须同时满足:
   ① trackable 是 Plane    ② Plane 处于 TRACKING
   ③ isPoseInPolygon(hitPose) = true  ← 落在已重建的平面多边形内

   ┌─────────────────┐
   │   Plane polygon  │   ✅ 命中点在多边形内 → 接受
   │      ● A         │
   │            ●? ───┼──→ ❌ 落在多边形外 → 拒绝(早期无限延伸误命中被挡掉)
   └─────────────────┘
```

### 用户怎么用 · 能测什么 · 哪种最准
- **用法**：对准有纹理的地面/桌面/墙面，缓慢移动到**准星变亮**（命中有效平面）才点 `+`。
- **测什么**：同一大平面上的两点（如地面通道宽度）。
- **哪种最准**：A4 纸平铺桌面这类「两点确实都在一张大平面多边形内」的情形——实测 A4 ≈29cm ✅。
- **📱 手机怎么动**：同方案二；且因只认 in-polygon 平面点，第二点所在的平面多边形必须**已被 ARCore 重建到那里**——常需**缓慢平移并稍作停留**等平面「长」过去、准星变亮再点。仍**不建议原地旋转**。

### 优点
- 排除特征点噪声，命中更稳；距离冻结消除跳变；**截图数字 == 台账数字**（同源）。
- 仅在点击瞬间门控，不每帧拦截 → 放点成功率不受影响（吸取了上一轮「过严门控被整体回退」的教训）。

### 劣势（这一版暴露了致命物理问题）
- **MacBook 宽度量出 97cm**：MacBook 本体没平面，射线穿过它落到**后方桌面**，两点都在桌面但中间隔着物体 → 距离虚高。
- 黑色垂直面量出 83cm 且漂移：低纹理/黑色表面 ARCore 跟踪差。
- 没有「两点必须在同一平面」的判定，跨平面也会出假数值。

### 为什么被取代
出现「看似成功、实则命中了错误平面」的假完成值，必须先**看清每次命中到底打到哪**。→ 诊断模式（方案四）。

## 方案四：诊断模式——同根平面硬拒绝（Step 7.x）

📅 2026-06-24 ｜ 「先诊断，不盲修算法」

### 设计
不改算法，先**加可观测性**：Debug Overlay（屏幕 `tv_debug` 面板）+ 每次点击打印完整 `[AR_MEASURE_DEBUG]` 日志（命中平面类型、是否 in-polygon、planeId 代理、hitPose/cameraPose、dx/dy/dz、距离…）。唯一行为变更：**同根平面硬拒绝**——终点与起点不在同一根平面（沿 `subsumedBy` 合并链找 rootPlane，用 `identityHashCode` 作代理标识比对）时，**不创建终点、不出距离**，提示「两个测量点未命中同一稳定平面」。

### 数学原理
用「根平面比对」近似「是否同一物理平面」，并图解 MacBook 97cm 根因：

```
  相机 ◇
        \   \
         \    \  第二条射线穿过 MacBook 本体
          \     \
  桌面 ─────●─────■■MacBook■■─────●───   ← 两点都在桌面平面,
           A                      B'       但 B' 在物体后方 → 距离虚高 97cm
  日志会显示: planeType=HORIZONTAL_UPWARD, samePlane=true, hitDistanceToCamera 偏大
```

### 用户怎么用 · 能测什么 · 哪种最准
- **用法**：同方案三；开发者另开 `adb logcat -s ArMeasurement | grep AR_MEASURE_DEBUG` 看诊断。
- **测什么 / 最准**：与方案三相同（A4 平面内两点）；本质是诊断层，不改变可测范围。
- **📱 手机怎么动**：同方案三；且第二点须落在与第一点**同一根平面**，所以只能**沿同一张平面平移**，不能转头去对另一面墙 / 另一块地（否则触发「未命中同一稳定平面」硬拒绝）。

### 优点
- 把「命中到底打到哪」完全暴露，定位了 MacBook 错误是**物理命中问题**而非算法 bug。
- 同根平面硬拒绝挡掉了跨平面假完成值。

### 劣势
- 「必须重新命中同一根平面」这条规则，为后续 v2 第二点**几乎永远失败**埋下伏笔。
- 解决不了「两点同在桌面、但中间隔着要测的物体」这类场景。

### 为什么被取代
诊断结论是：要测真实物体，需要**更强的命中能力（Depth）+ 更几何化的同平面判定**。→ v2 重构（方案五）。

## 方案五：v2 重构——Depth + 候选评分 + 同物理平面一致性

📅 2026-06-24 ｜ 引入多个新 Kotlin 模块

### 设计
系统化重构：
- **Depth API** 采样命中点深度（`ArDepthSampler`），扩展非平面覆盖；
- **候选评分** `ArHitScorer`：对一次 hitTest 的多个候选打分选最优（落在 polygon 外 `score -= 1000` 等）；
- **同物理平面一致性** `ArPlaneConsistency`：不再比对「同一 Plane 实例」，改用**几何量**判定终点与起点是否共面——法线夹角、点到起点平面距离（容差 3cm）、深度差（容差 5cm）、表面类型一致；
- 锚点漂移监测 + 距离冻结。

### 数学原理
同物理平面判定的三个几何量：

```
① 法线夹角:   θ = arccos( (n_start · n_end) / (|n_start||n_end|) )   ≤ 阈值
② 点到平面距离: dist = |(P_end − P_start) · n_start| / |n_start|      ≤ 3cm
③ 深度差:     |depth_start − depth_end|                              ≤ 5cm

      n_start ↑
              │      P_end
   ───────────●───────●───   起点拟合平面
            P_start    ╲
                        ╲ dist = P_end 到该平面的垂直距离
```

平面法线取自 `plane.centerPose` 旋转矩阵第三列 `(m8,m9,m10)`。

### 用户怎么用 · 能测什么 · 哪种最准
- **用法**：同相机式 UI，点两次 `+`。
- **测什么（理论）**：A4 / MacBook / 黑色垂直面。
- **哪种最准（理论）**：纹理好、两点严格共面时。
- **📱 手机怎么动**：理论同方案三/四（两点都 hitTest、靠平移视差 + 命中平面）；但第二点被 polygon `-1000` 与 3cm/5cm 容差拦死，**实际上无论怎么平移或旋转都很难放上第二点**——这正是它被废弃的原因。

### 优点
- Depth 扩展了可命中表面；评分能避开明显劣质候选；一致性用几何量比「同一实例」更鲁棒。

### 劣势（致命）
- 终点仍须 `hitTest` 重新命中 + 过评分 + 过一致性校验。小物体（A4、桌上书本）的平面 polygon 不够大，或 ARCore 在第二处生成了**另一个** Plane → **polygon 外 `-1000`** 直接否决，叠加 3cm/5cm 容差 → **第二个点几乎一次都放不上**。
- 真机表现：第一点容易加，第二点永远被「未命中同一稳定平面」拦死。

### 为什么被取代
「第二点必须重新命中同一稳定平面」这条策略**根本不可用**。必须彻底放弃「第二点再 hitTest」。→ v3（方案六）。

## 方案六：v3——锁定平面 + 屏幕射线投影（当前方案）

📅 2026-06-24 ｜ 新增 `ArPlaneRayIntersector.kt`，改 `ArMeasurementActivity.kt` / `ar_strings.xml`

### 设计
状态机简化为 `IDLE → PLANE_LOCKED → MEASURED`：
- **第一点**：屏幕中心 `hitTest` 命中稳定 Plane → 创建起点锚点，并**锁定**这个平面（记录平面上一点 + 法线 + 类型，存入 `startSnapshot`）；
- **第二点**：**不再 hitTest**。取相机当前位姿算出屏幕中心世界射线，与「第一点锁定的**无限延伸**平面」求交，得到终点世界坐标；
- 距离 = 两锚点世界坐标欧氏距离，终点创建瞬间冻结。

### 数学原理
两步几何：屏幕中心射线 + 射线-平面求交。

```
        相机光心 O (rayOrigin)
            \
             \   rayDir d = 屏幕中心(准星)方向
              \      （由 inverse(proj×view) 反投影 NDC(0,0) 的近/远点求得）
               \
   ─────●───────●──────────  ← 第一点锁定的无限平面 (平面上一点 P0, 法线 n)
       A         X = endWorld
     (起点)      = 射线 ∩ 平面

   求交:  t = ((P0 − O) · n) / (d · n)         (d·n≈0 → 平行,无解; t≤0 → 平面在背后)
          X = O + t · d
   距离:  distance = ‖X − A‖
```

**为什么这样算是对的**：本模式专做「平面内两点测距」，第二个端点**物理上本就在锁定平面上**；你把准星对准它，从相机穿过准星的视线必然在该端点的真实位置穿过这个平面，所以交点 = 端点真实世界坐标——**无需 ARCore 在那里重建任何几何**。两点共面，连线即平面内真实长度。

### 用户怎么用 · 能测什么 · 哪种最准
- **用法**：准星对准被测物所在的**支撑平面**（桌面/地面），变亮后点 `+` 锁面放起点 → 准星移到另一端点 `+` 投影出终点。
- **测什么**：**平面内两点距离**——A4 纸长度、桌面书本长度、纸箱**底边**宽度、消防通道地面净宽。
- **哪种最准**：两个端点**确实都贴在同一平面上**、俯视角拍摄时最准（A4 ≈29.7cm ±2cm）。
- **📱 手机怎么动（直接回答「能不能移动 / 要不要固定旋转」）**：**平移和原地旋转都支持，手机可以移动**。因为第二点不靠 hitTest，而是用「**当前相机位姿** + 屏幕中心射线 + 锁定的无限平面」求交——相机移到哪、朝哪都行，只要准星射线能打到锁定平面。
  - ✅ **最稳**：锁面后**小幅平移**让准星滑到第二端点，尽量**俯视**（射线接近垂直打平面），全程缓慢、保持 TRACKING、画面有纹理。
  - ⚠️ **原地旋转也行但要小心**：站定只旋转手机把准星甩到第二点，数学上完全成立；但 (1) 单目 ARCore 纯旋转时位姿/尺度易漂，位姿一偏交点就偏；(2) 看较远的点时射线与平面接近**掠射**（`d·n`→0），微小位姿误差被放大成大的位置误差。所以原地旋转量近处小物体（A4）尚可，量远处 / 大跨度会变差。
  - ❌ 别快速大幅挥动（丢 TRACKING 一切白搭）。

### 优点
- 第二点**不再依赖** ARCore 在该处重建几何 / 同 planeId / in-polygon → 彻底解决「第二点放不上」。
- 两点强制共面 → 距离几何自洽、可信。
- 全程世界坐标，绝不用屏幕像素/UI 连线长度/缩放系数。

### 劣势
- **唯一假设**：第二端点必须在锁定平面上。一旦离面（厚书顶面、纸箱**高度**），会被投影回平面产生误差；斜视角放大该误差。
- 仅支持平面内两点：**不支持**高度、空中两点、非平面边缘、任意 3D、多段测量。
- 低纹理/纯白桌面第一点平面识别仍慢。

### 为什么是当前终点
它用「锁定平面 + 投影」同时满足了两个一直打架的目标——**第二点放得下**（不依赖重建几何）且**距离算得准**（两点共面）。在「平面内两点」这个被刻意收窄的范围内，这是稳定可用的最简方案。

## 📱 拍摄时手机到底怎么动？（横向对比 + 直接回答）

> 核心疑问：测距时手机**能不能移动**？还是必须**固定在一个点、靠旋转手机**把准星甩到第二个锚点？

答案取决于「第二个点怎么拿到世界坐标」：

| 方案 | 第二点怎么定位 | 手机能否移动 | 推荐姿势 | 原地旋转可行？ |
|---|---|---|---|---|
| 1 像素估算 | 单帧，无第二点 | 端稳即可 | 稳住、正对物体 | 不涉及 |
| 2–5 hitTest 系 | 第二点必须**真实 hitTest 命中平面** | **必须移动**（平移产生视差 + 让平面覆盖到第二点） | 缓慢小幅平移 | ❌ 单目纯旋转退化 |
| 6 v3 投影（当前） | 第二点 = 当前位姿射线 ∩ 锁定平面 | **可平移、也可旋转** | 小幅平移 + 俯视最稳 | ⚠️ 可以，但远 / 掠射会变差 |

### 针对当前 v3 的直接结论
- 你**可以移动手机**，不必死守一个点。
- **最准的做法**：锁定第一点后，**小幅平移**让准星滑到第二个端点，尽量**从上往下俯视**（射线垂直打到平面）。
- **原地旋转（站定不动只转手机）也能用**，数学上完全支持；但单目 ARCore 纯旋转时跟踪容易漂，而且对着远处会「掠射」放大误差——量 A4 这种**近处小物体没问题**，量大跨度建议**走近 + 平移**。
- **共同红线**：全程缓慢、保持画面有纹理、别快速大幅挥动（任何方案丢了 TRACKING 都白搭）。

### 为什么 hitTest 系（2–5）必须平移、不能原地旋转
单目 SLAM 靠相机**平移产生的视差**来三角化深度和尺度；原地纯旋转没有视差，尺度会漂。而且第二点要 hitTest **真实命中**，那个位置得有已重建好的平面，原地旋转往往覆盖不到。v3 把第二点改成「投影到已锁定的无限平面」后，才解除了「必须平移命中」的限制——于是旋转也行了，这是 v3 体验更自由的一个副产品。

## 一条主线：演进逻辑总结

1. **从「猜深度」到「测两点」**（方案 1→2）：消防要的是净宽/门高这类两点实距，像素估算给不了。
2. **从「自由命中」到「只认平面」**（方案 2→3）：命中太自由 → 结果不可信，收紧到 in-polygon 平面点。
3. **从「盲修」到「先看清」**（方案 3→4）：加诊断暴露出「MacBook 97cm 是射线穿物体落桌面」，确认错在物理命中而非算法。
4. **从「同实例」到「几何一致性 + Depth」再到发现死角**（方案 4→5）：试图用更强命中 + 几何校验覆盖全场景，却让第二点被严格校验拦死。
5. **从「第二点重新命中」到「第二点投影到锁定平面」**（方案 5→6）：**这是关键转折**——放弃「第二点必须自己 hitTest 命中同一平面」，改成「第一点锁定无限平面、第二点只做射线投影」，一举解决放不上点 + 保证共面准确。

**核心教训**：测距的难点从来不是「求两点距离」（固定公式），而是「**第二个点怎么可靠地变成一个与第一个点可比的世界坐标**」。v3 的答案是：**不要让第二个点去依赖环境重建，而是把它约束到第一个点已经锁定的几何上**。

## 适用场景速查（当前 v3）

✅ **能测准**：A4/纸张长宽、桌面上书本/文件长度、纸箱**底边**宽度、地面消防通道净宽——凡是「两端都贴在同一支撑平面上」的。

⚠️ **有误差**：被测物离支撑面较高（厚书顶面）、斜视角。

❌ **不支持**：纸箱**高度**、空中两点、非平面物体边缘、任意 3D 尺寸、多段折线测量。
