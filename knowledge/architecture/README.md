# knowledge/architecture

## Purpose
Reference material on system architecture, design patterns, and high-level technical decisions.

## What Belongs Here
- Architectural Decision Records (ADRs).
- System design patterns and trade-off analyses.
- Technology selection criteria and comparisons.

## What Does NOT Belong Here
- Low-level implementation details (use `backend/` or `engineering/`).
- Project-specific docs (use `projects/`).

## Lifecycle
1. Document significant architectural decisions.
2. Review and update as the system evolves.
3. Reference during new project planning.

## Example Entry
- `adr-014-event-sourcing.md`: An ADR documenting the decision to adopt event sourcing for the billing service.
