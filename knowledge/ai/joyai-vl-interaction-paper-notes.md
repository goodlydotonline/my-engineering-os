## 1. 概述

这篇技术报告提出并开源了 **JoyAI-VL-Interaction**，一个面向“实时视觉-语言交互”的 8B 参数模型及其完整可部署系统。它的核心目标不是让模型“被问到才回答”，而是让模型像一个真正在场的人一样，持续注视正在发生的视觉流，自己判断什么时候该说话、什么时候该保持沉默、什么时候又该把复杂任务交给更强大的后台模型处理。

作者认为，当前主流大模型和多模态产品本质上仍是“轮询式”的：无论延迟多低、架构多原生，它们都只在用户发起一轮对话后才睁眼并作答。对于火灾监控、老人跌倒、直播抢购、AI 眼镜实时翻译这类场景，关键时机往往在人提问之前就已经过去。因此，模型必须具备“主动性”（proactivity）、“时间感知”（time awareness）和“长程记忆”（long-horizon memory）三种能力，并把“何时行动”作为模型自身要学习的一项核心能力。

论文的两个主要贡献是：

1. **JoyAI-VL-Interaction 模型**：一个约 8B 参数、以视觉为首要驱动模态的交互模型。它每秒钟对输入视频做一次决策，输出三种动作之一：`<silence>`（保持沉默）、`<response>`（回复用户）或通过 `<delegation>`（委托后台模型）。论文同时开源了训练配方和数据构造方法。
2. **完整可部署系统**：围绕该模型构建的开源系统，支持 WebRTC/RTSP 视频流输入、可替换的 ASR/TTS、长程记忆、后台模型桥接、可视化 UI 以及基于 vLLM 的实时推理，目标是“开箱即用”。

## 2. 问题定义

### 2.1 现实世界不等用户提问

许多需要 AI 介入的时刻具有强烈的“时间敏感性”：

- 监控画面里突然出现火焰；
- 视频通话中对方表情闪过；
- 直播里心仪商品一闪而过；
- 老人独自在家摔倒。

这些事件的共同点是：值得回应的瞬间在用户组织语言之前就已经出现，错过就是错过。传统的“问答式”模型无法主动发现这些时机。

### 2.2 现有方案为何不够

作者把现有方案分为三类，并指出它们各自的局限：

1. **实时/全模态模型**（如 GPT-Realtime-2、Qwen3.5-Omni）：确实是端到端低延迟模型，但优化目标仍是“用户说完后尽快回答”，核心仍是回合制对话。
2. **消费级视频通话产品**（如 Doubao、Gemini 的应用内视频通话）：Doubao 通过每隔几秒触发一次后台请求来模拟主动，但本质上还是“外部定时器 + 轮询”，事件响应延迟不可能低于轮询间隔；Gemini 则基本保持严格的“一问一答”。
3. **流式视频理解研究**：多停留在实验室，通常只解决响应速度、主动性、长程记忆中的一个维度，且缺乏完整的部署系统。

### 2.3 交互模型（Interaction Model）的定义

论文采用 Thinking Machines Lab 提出的“交互模型”概念，并把它形式化：

> 交互模型每一时刻都自主判断“是否要对该视觉流采取行动”，包括无人提示时主动开口；而轮询模型只能在用户的回合到达时行动。

因此，关键差异不是“快不快”，而是“能不能自己决定何时行动”。

## 3. 方法解析

### 3.1 模型架构：以视觉为第一驱动

JoyAI-VL-Interaction 基于 **JoyAI-VL 1.0** 进行后训练：

- **语言模型**：初始化自 Qwen3-8B；
- **视觉编码器**：Qwen3-VL ViT（约 400M 参数）；
- **投影层**：从头训练，用于桥接视觉和语言模态。

JoyAI-VL 1.0 已经通过表征对齐、视觉-语言预训练以及基于 On-Policy Distillation 和 RL 的后训练，成为一个常规的“回合制”VLM。论文在此基础上，通过交互式后训练把它变成“流式交互模型”。

#### 视频编码：AdaCodec

为了让模型能持续“在线”而不被无限增长的视频帧拖垮，论文使用 **AdaCodec** 进行预测性视觉编码：

- 只在“参考帧”（reference frame）上花费完整的 ViT token（约 256 个）；
- 对预测误差小的中间帧，用运动信息和残差压缩成约 16 个 **P-token**；
- 当场景变化大到难以预测时，自动插入新的参考帧。

这样，视频流的成本随“场景变化量”增长，而不是随帧数线性增长，使得模型能在普通 vLLM 推理基础设施上持续运行数小时。

### 3.2 每秒三动作：沉默 / 回复 / 委托

模型每秒钟对当前视觉流做一次离散决策：

| 动作 | 含义 | 输出示例 |
|------|------|----------|
| `<silence>` | 继续观察，不说话 | `</silence>` |
| `<response>` | 在当前时机给出语言回复 | `</response> 火焰出现了。` |
| `<delegation>` | 把复杂子任务交给后台模型异步处理 | `</delegation> 解释 boat hoist 的受力结构……` |

其中“沉默”被当作一等动作来学习，而不是“没有标签”。这一点很重要，因为流式视频中绝大多数时刻其实都不需要说话。

### 3.3 数据构造：时间对齐的交互数据

论文构建了超过 **400 万条时间对齐的流式视频片段**，分为六大类：

1. **主动告警与异常检测**：火灾、冲突、入侵等；
2. **时间对齐问答**：包括“过去、现在、未来”三种时态——证据已出现、正出现、未出现；
3. **计数与时序感知**：数飞镖、数过人次数等；
4. **实时解说与旁白**：体育赛事、宠物直播等；
5. **多轮闲聊**：围绕视频内容进行自然对话；
6. **委托任务**：把 STEM 问题、视频推理等难题交给后台模型。

#### 数据构造的关键原则

每个样本必须在两条轴上都正确：

- **内容轴**：说什么；
- **时间轴**：在第几秒说、保持沉默还是委托。

论文采用多阶段 verifier 流水线：对每个候选样本做“全局检查”（整段视频 + 完整标注）和“局部检查”（标注时刻附近的帧 + 对应回复），只有全部通过才进入训练集。不同类型的数据来源会针对其“瓶颈轴”做重点标注：

- 离线视频 QA 数据内容可靠，重点做时间定位；
- 闲聊数据时间要求宽松，重点把每轮对话锚定到具体时间点；
- 解说/广播数据用 ASR 还原真实人声，保留自然的停顿与沉默；
- 告警数据通过 onset 检测，确保警报只标在事件“第一次出现”的帧。

#### 委托任务的“双循环”训练

为了让模型学会“在等后台答案时仍然保持在线”，委托数据会模拟随机延迟：模型先给用户一个简短回应（如“让我查一下”），再发出隐藏委托请求；后台结果在若干秒后才返回。训练时模型必须在等待期间继续观察、继续对话、保持沉默，从而把实时循环和异步循环缝合起来。

### 3.4 训练配方

#### 继续训练（Continue Training）

把 3.3 节的时间对齐数据混入大量常规回合制数据，对 JoyAI-VL 1.0 做监督微调。作者发现，只需加入相对少量的时间对齐数据，交互行为就会涌现。

#### 加权交叉熵损失

时间对齐数据中 `<silence>` 占绝大多数，标准 SFT 会压倒性地倾向于“一直沉默”。论文对控制 token 做加权：

- 首次 `<silence>`：权重 1；
- 连续重复的 `<silence>`：权重降到 0.4；
- `<response>`：权重提升到 1.5。

公式形式为归一化加权交叉熵：

```
L(θ) = - (1/|A|) Σ_{j∈A} w_j · log p_θ(y_j | y_{<j})
```

#### 强化学习阶段

继续训练能学会三种动作，但“时机”这种细粒度能力很难靠 token 级监督完美掌握。论文使用 **GRPO**（Group Relative Policy Optimization）直接优化流级奖励：

- 奖励：正确且时机恰当的回复、恰当的沉默、合理的委托；
- 惩罚：无故开口、时机错误、总是回复等退化行为；
- 为控制 rollout 长度，使用 **answer-centered window sampling**：只保留与时机决策相关的少量回合，把原本数百步的静默流压缩成可训练的短轨迹；
- 回复内容还会用 LLM judge 按任务 rubric 评分，作为细粒度奖励。

训练框架使用 **EasyVideoR1**，一个面向视频-语言模型强化学习的高效框架。

### 3.5 系统架构：可部署的完整栈

系统设计的核心原则是：**只有交互模型决定“何时说/何时委托”，其余所有模块都可插拔。**

#### 双循环：看见与行动

系统同时运行两条循环：

- **实时循环**：持续接收 WebRTC（浏览器摄像头）或 RTSP（直播/监控）流，默认以 1 Hz 采样，把帧转成 Base64 图片后输入模型；可选的麦克风语音通过流式 ASR 转成文本查询；模型每秒输出动作，TTS 实时播报回复。
- **异步循环**：当模型输出 `<delegation>` 时，系统把一个带任务 ID、问题描述、上下文帧快照的文本请求发给“后台大脑”（可以是任意大模型、Agent 或 API）。后台异步推理期间，模型继续与用户保持实时互动；结果返回后再自然地把答案织入对话。

这种设计让“看见物理世界 → 判断 → 在数字世界行动”的链路闭合，形成论文所谓的 **watch-and-do** 模式。

#### 可插拔模块

| 模块 | 默认实现 | 可替换为 |
|------|----------|----------|
| ASR/TTS | FunASR / CosyVoice | 任意 ASR/TTS API 或模型 |
| 可视化 UI | 基于 live-vlm-webui | 自定义前端 |
| 后台大脑 | 用户自己的大模型 API | Agent 系统（如 Hermes Agent、OpenClaw） |
| 记忆 | 三层分层记忆 | 任意外部记忆实现 |
| 推理服务 | vLLM | 标准 vLLM 生态 |

ASR/TTS 不参与决策。为避免语音“堆积”，规则是：若上一句还在播放，下一句只显示文本、不合成语音。

#### 长程记忆

为支持数小时连续视频，系统采用可插拔的三层分层记忆：

1. **短期记忆**：保留最近 `T_s` 秒（默认 100 秒）的原始视觉 token；
2. **中期记忆**：最多 `M` 段文本摘要，覆盖 `M × T_s` 秒；
3. **长期记忆**：最多 `L` 个高度压缩的块，每块由 `M` 段中期摘要聚合而来，覆盖 `L × M × T_s` 秒。

默认配置 `T_s=100, M=5, L=15` 可支持约两小时上下文。由于中层以上是文本，可被 vLLM 的 prefix cache 缓存复用，避免每步重算全历史。

#### vLLM 原生推理

论文强调，很多流式视频方法因为无法利用 vLLM/SGLang 的 prefix cache 而只能停留在实验室。JoyAI-VL-Interaction 的记忆结构围绕 prefix reuse 设计：每段文本记忆预先填充进 KV cache，后续每步只计算新帧和最新回复。配合 AdaCodec 把可预测帧压到约 16 token，系统可在标准 vLLM 上以 **亚秒级延迟** 持续运行 **超过两小时**。

## 4. 实验与结果

### 4.1 评测设置

论文没有采用传统离线视频理解 benchmark，而是直接对比两个成熟消费级产品：**Doubao** 和 **Gemini** 的应用内视频通话助手。

评测场景共 6 个、58 个用例：

1. 监控与告警（10 例）
2. 实时计数（10 例）
3. 实时翻译（10 例）
4. 时间感知（10 例）
5. 实时解说与引导（9 例）
6. 长程记忆（9 例）

评分方式：5 位具备 LLM 研究背景的人类评分员，对每个用例从 **质量**（正确、相关、通顺）和 **时机**（不早不晚、该沉默时沉默）两个维度打分，各分 good/fair/poor 三档，再取等权平均。最后统计 JoyAI-VL-Interaction 相对基线的 **胜率 / 平率 / 负率**。

### 4.2 主要结果

**对阵 Doubao：**

| 场景 | JoyAI 胜率 | 平率 | Doubao 胜率 |
|------|-----------|------|------------|
| 监控与告警 | **100.0%** | 0.0% | 0.0% |
| 实时计数 | **70.0%** | 30.0% | 0.0% |
| 实时翻译 | **80.0%** | 20.0% | 0.0% |
| 时间感知 | **80.0%** | 10.0% | 10.0% |
| 实时解说 | **55.6%** | 22.2% | 22.2% |
| 长程记忆 | **77.8%** | 22.2% | 0.0% |
| **总体** | **77.6%** | 17.2% | 5.2% |

**对阵 Gemini：**

| 场景 | JoyAI 胜率 | 平率 | Gemini 胜率 |
|------|-----------|------|------------|
| 监控与告警 | **100.0%** | 0.0% | 0.0% |
| 实时计数 | **100.0%** | 0.0% | 0.0% |
| 实时翻译 | **100.0%** | 0.0% | 0.0% |
| 时间感知 | **50.0%** | 40.0% | 10.0% |
| 实时解说 | **100.0%** | 0.0% | 0.0% |
| 长程记忆 | **77.8%** | 22.2% | 0.0% |
| **总体** | **87.9%** | 10.3% | 1.7% |

### 4.3 结果解读

- **时间敏感型任务优势最大**：监控告警、实时计数、实时翻译对 JoyAI 几乎都是碾压级优势。这正验证了“主动时机判断”是这类任务的核心瓶颈。
- **长程记忆优势明显**：部分用例中 Doubao/Gemini 会在无语音输入约 2–5 分钟后自动挂断，导致根本不在场；JoyAI 的分层记忆使其在数小时后仍能回答早期内容。
- **实时解说相对胶着**：Doubao 在解说场景赢得 22.2%，主要依靠更大模型带来的知识量和文采；但时机上仍不稳定。JoyAI 的失利主要来自偶发的幻觉，作者归因于模型规模和解说数据相对稀疏。
- **时间感知 vs Gemini 最接近**：因为部分时间感知用例是“事后再问”，对实时性要求不高，Gemini 的强基础模型能靠质量追上。

### 4.4 代表性案例

论文列举了 6 个案例（完整 side-by-side 视频见项目博客）：

1. **跌倒检测**：人摔倒瞬间 JoyAI 立即告警；Doubao 延迟 4–5 秒；Gemini 直接表示无法监控。
2. **飞镖计数**：6 支飞镖命中时 JoyAI 实时累计；Doubao 只数到 2 支且延迟；Gemini 只回了一句“让我看看”。
3. **街头采访字幕翻译**：JoyAI 持续翻译屏幕字幕；Doubao/Gemini 只在被问的那一刻翻译当前帧，然后停止。
4. **20 秒计时**：JoyAI 误差 1–2 秒；Doubao 未响；Gemini 约 40 秒才响。
5. **宠物直播解说**：13 段短片中 JoyAI 能跟上并描述每只宠物的情绪；Doubao 只描述了 3 段且不准确；Gemini 只说一句。
6. **手机界面委托**：要求根据屏幕生成 HTML，JoyAI 判断超出实时能力，委托后台模型并返回可用 HTML。

### 4.5 涌现能力

作者还展示了两个训练数据中从未出现过的能力：

- **购物 App 引导**：用户在手机上滑动购物，JoyAI 能跟随每个界面变化并主动引导找到目标商品；训练数据中没有任何 App 界面视频。
- **旅行场景定时解说**：被要求每 4 秒解说一次，JoyAI 能稳定保持节奏且内容扎实；训练数据中没有“定时 + 解说”组合任务。

作者把这视为模型学到了某种通用的“看-交互”能力，而不只是记忆了特定场景脚本。

## 5. 讨论与启发

### 5.1 定位与范式意义

JoyAI-VL-Interaction 试图推动 VLMs 从“问答工具”走向“在场的协作者”。这不仅是功能增强，而是交互范式的转换：从“提交请求并等待”到“watch-and-do”。这种能力对 AI 眼镜、视障辅助、老人/儿童看护、家庭机器人、直播监控等场景都有直接价值。

### 5.2 局限性

论文对自身局限很坦诚：

1. **规模较小**：8B 模型在通用知识、聊天润色、复杂输入鲁棒性上不如 Doubao/Gemini 背后的大模型。
2. **数据尚处早期**：数据配比、规模、清洗都还有优化空间，解说场景的偶发幻觉与此直接相关。
3. **评测范围有限**：仅 6 个场景、58 个用例、两个基线，且依赖人工评分，规模化和客观化都不够。
4. **语音并发处理较粗**：高频连续输出场景建议关闭 TTS，更精细的音频打断/重叠处理留待未来。

### 5.3 对未来工作的启发

- **交互能力可独立 scaling**：论文暗示“何时行动”是一种与“智力”并行的 scaling 维度。即使模型不大，只要在交互维度上专门训练，也能在实时场景打败大得多的模型。
- **数据格式比领域更重要**：统一的“每秒一动作”格式让 heterogeneous 数据来源能被训练同一个策略，也说明该配方可以迁移到新领域。
- **系统与模型必须一起开源**：论文把模型、数据配方、完整系统同时开源，这对社区复现和扩展非常关键。
- **委托机制是小型模型对抗大模型的关键**：通过把重推理交给后台，8B 模型既保持实时在场，又能调用任意规模的能力。

## 6. 术语附录

| 术语/缩写 | 解释 |
|----------|------|
| **VLM**（Vision-Language Model，视觉-语言模型） | 能同时理解图像/视频和文本的多模态大模型。 |
| **VL-Interaction**（视觉-语言交互模型） | 本文提出的模型类型：持续注视视觉流并自主决定何时说话的模型。 |
| **Turn-based**（回合制） | 模型只在用户发起一轮对话后作答，其余时间空闲。 |
| **Proactivity**（主动性） | 模型在无人提示时也能根据视觉事件主动开口。 |
| **Time awareness**（时间感知） | 模型对“现在、过去、未来”以及经过时间的感知能力。 |
| **AdaCodec** | 论文使用的预测性视觉编码器，用参考帧 + P-token 压缩视频。 |
| **P-token** | AdaCodec 中对可预测帧的紧凑表示，约 16 个 token。 |
| **ViT**（Vision Transformer） | 视觉 Transformer，用于把图像/视频帧编码为 token 序列。 |
| **Token / Tokenization** | 模型处理的基本单元；视觉 token 即图像/视频经过编码后的向量序列。 |
| **ASR**（Automatic Speech Recognition，自动语音识别） | 把语音转成文本。 |
| **TTS**（Text-to-Speech，文本转语音） | 把文本转成语音。 |
| **WebRTC** | 浏览器实时音视频通信协议，用于摄像头输入。 |
| **RTSP**（Real Time Streaming Protocol） | 网络摄像头/直播流常用协议。 |
| **vLLM** | 面向 LLM/VLM 的高吞吐推理引擎，支持 KV cache 和 prefix caching。 |
| **KV Cache / Prefix Reuse** | 把已计算好的注意力键值缓存起来，让后续请求复用，降低延迟。 |
| **GRPO**（Group Relative Policy Optimization） | 一种强化学习算法，通过组内相对奖励优化策略。 |
| **EasyVideoR1** | 论文使用的视频-语言模型强化学习训练框架。 |
| **SFT**（Supervised Fine-Tuning，监督微调） | 用标注数据直接微调模型。 |
| **RL**（Reinforcement Learning，强化学习） | 通过奖励信号优化模型策略。 |
| **Background Brain / 后台大脑** | 接收模型委托的难任务并异步返回结果的外部模型/Agent/API。 |
| **Delegation**（委托） | 模型把当前无法实时处理的子任务交给后台大脑。 |
| **Three-tier Memory**（三层记忆） | 短期原始视觉 token、中期文本摘要、长期压缩块的层次化记忆。 |
| **Omni Model**（全模态模型） | 同时处理文本、图像、音频、视频等多种模态的模型。 |
| **Streaming Video**（流式视频） | 视频数据边产生边处理，而不是等整个视频下载完再处理。 |
| **Watch-and-do** | 论文提出的交互范式：模型主动观察世界并据此行动。 |
