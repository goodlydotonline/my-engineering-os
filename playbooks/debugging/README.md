# playbooks/debugging

## Purpose
Procedural guides for diagnosing and resolving common technical issues.

## What Belongs Here
- Step-by-step debugging checklists.
- Log analysis and tracing procedures.
- Common error patterns and resolutions.

## What Does NOT Belong Here
- Theoretical background (use `knowledge/`).
- Project-specific operational docs (use `projects/`).

## Lifecycle
1. Document a debugging session as a repeatable procedure.
2. Test and refine the playbook.
3. Update after each significant incident.

## Example Entry
- `memory-leak-in-go-service.md`: A checklist for profiling and fixing memory leaks in Go services.
