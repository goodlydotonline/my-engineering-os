# rules/project-rules

## Purpose
Project-specific rules, constraints, and guardrails.

## What Belongs Here
- Domain-specific constraints (e.g., compliance, security).
- Project-level architectural constraints.
- Custom rules that override general standards for a specific project.

## What Does NOT Belong Here
- General coding standards (use `coding/`).
- AI workflow definitions (use `ai-workflow/`).

## Lifecycle
1. Define rules when a project has unique constraints.
2. Document and communicate to all contributors.
3. Review and update as project requirements change.

## Example Entry
- `payment-service-rules.md`: Security and compliance rules specific to the payment service.
